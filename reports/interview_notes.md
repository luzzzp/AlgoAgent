# AlgoAgent 面试讲解笔记

## 项目解决的问题

通用代码模型生成的算法答案经常看起来合理，但可能：

- 无法编译。
- 无法通过边界测试。
- 使用理论上会超时或超内存的算法。
- 根据测试反例反复修改后，对评测测试产生过拟合。
- 在代码尚未验证时仍然给出自信的解题讲解。

AlgoAgent 将算法题解题设计成一个无泄漏、可验证的闭环。

## 为什么区分修复测试和留出测试

修复测试用于向 Agent 提供真实反例，使模型能够修复语法和逻辑错误。

留出测试只用于最终评测。Agent 如果在留出测试失败，会直接返回 `FAILED`，不会看到具体输入、期望输出，也不会继续修复。这样可以避免用同一批测试既调试代码又计算最终效果。

## 为什么隔离 OracleMetadata

真实应用中通常不知道题目难度、算法标签、标准复杂度和标准答案。

因此运行时 Agent 只能访问：

- 题面。
- 输入输出格式。
- 数据范围。
- 语言。
- 时间和空间限制。

难度、标签、标准复杂度和参考答案仅用于训练数据构造与离线分组统计。

## 资源约束检查

模型必须输出时间复杂度和空间复杂度。系统从数据范围中提取变量上界，并按照 C++ 每秒约 `1e8` 次基础操作估算理论运行量。

资源检查结果允许为：

- `PASS`：估算未超过限制。
- `FAIL`：理论上会 TLE 或 MLE。
- `UNKNOWN`：无法可靠估算，不能伪装成通过。

复杂度分析是启发式判断，因此仍然保留真实编译和测试超时。

## Agent 工作流

```text
题面与资源限制
    ↓
生成代码与复杂度
    ↓
理论资源检查
    ↓
修复测试与反例反馈
    ↓
不可反馈的留出测试
    ↓
验证成功后生成解题思路
```

只有通过全部验证后，系统才返回代码和解题思路。否则返回明确的 `FAILED` 和诊断摘要，不返回未验证代码。

## 核心指标

- Initial Compile Rate
- Final Compile Rate
- Repair Test Pass Rate
- Held-out Test Pass Rate
- Verified Success Rate
- Avg Repair Turns
- Theoretical TLE Rate
- Theoretical MLE Rate
- Complexity Unknown Rate
- Failure Breakdown
- Explanation Success Rate
